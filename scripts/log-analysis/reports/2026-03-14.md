### Detailed Analysis

**`/v1/pipeline/auto-publish/{uuid}/status`** (69 slow requests)
- **Handler**: `pipeline_routes.py:154` — trivial PK SELECT
- **Bottleneck**: DB pool connection acquisition. During sync, pool connections are held for minutes. Status polls wait up to 30s at pool_timeout.
- **Post-reinit behavior**: Still slow (10-22s) even with pool_size=3/2. The sync background task holds connections during Google Drive API + AI processing, starving status polls.
- **Recommendation**: Separate read-only pool for status endpoints, or cache task status in-memory.

**`/v1/worklist/sync`** (30 slow requests, all 202)
- **Handler**: `worklist_routes.py:106` — designed to return 202 immediately
- **Bottleneck**: Phase 1 DB commit (lines 130-138) waits for pool connection before creating PipelineTask record. The `asyncio.create_task` fix works correctly — the sync runs in background — but the initial INSERT blocks on pool acquisition.
- **Recommendation**: Use `asyncio.wait_for(session.commit(), timeout=5)` for the Phase 1 INSERT.

**`/health`** (1 slow request, 15.0s)
- **Handler**: `main.py:93` — `SELECT 1` health check
- **Bottleneck**: Cold start (container boot + Playwright imports) and/or pool connection wait.
- **Recommendation**: Bypass connection pool for health checks; use direct asyncpg connection.
